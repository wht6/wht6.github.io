请求头

HTTP请求头包含三个部分：请求行、请求头和请求体。

这里的第一行是请求行，中间是请求头，即header，最后是请求体body。

①是请求的类型，请求的类型包括以下几类：

GET、POST、PUT、DELETE、PATCH、HEAD、OPTIONS、TRACE

最常用的是GET和POST请求，RESTful api会用GET、POST、DELETE、PUT。

②是URL字段。与headers的Host字段组合就是完整的URL地址。

③协议和版本号。

④是请求头，包含的字段和对应的意义如表

Accept 浏览器可以接受服务器回发的数据类型
Referer 请求的来源，指示是从哪个URL跳转过来的
Accept-Language 浏览器可以接受的语言
User-Agent 客户端使用的操作系统和浏览器的名称和版本
Content-Type 内容资源的类型和字符编码
Host 主机名，即域名和端口
Content-Length 内容长度
Connection 连接类型，keep-alive长连接，close一次性连接。
Cache-Control 针对响应，默认private，私有缓存；public，缓存可多用户共享；no-cache，不缓存；max-age=10，缓存最大有效时间10s
Cookie 用于网站辨识用户身份

⑤是请求体，主体，是一个可选的数据块。

响应头

http响应头的格式：

第一行是状态行：包含HTTP协议版本、状态码和状态消息。

然后是响应头部分：描述响应信息。

响应头之后是一个空行。

空行后面是响应体：响应体包含的是实体内容，可以是html、json、图片等。

响应头包含的字段：

Content-Type：内容类型,告知客户端实体主体是什么格式
Content-Length：内容长度,告知实体主体长度
Content-Encoding：内容编码,告知客户端主体使用的编码
Cache-Control：缓存控制,控制缓存行为
Expires：过期时间,告知缓存的过期时间
Last-Modified：最后修改时间,告知客户端内容的最后修改时间
ETag：内容标识,用于缓存验证
Server：服务器名称和版本号信息

2.0和1.x对比

HTTP2.0和HTTP1.X相比的新特性

  • 新的二进制格式（Binary Format），HTTP1.x的解析是基于文本。基于文本协议的格式解析存在天然缺陷，文本的表现形式有多样性，要做到健壮性考虑的场景必然很多，二进制则不同，只认0和1的组合。基于这种考虑HTTP2.0的协议解析决定采用二进制格式，实现方便且健壮。
  • 多路复用（MultiPlexing），即连接共享，即每一个request都是是用作连接共享机制的。一个request对应一个id，这样一个连接上可以有多个request，每个连接的request可以随机的混杂在一起，接收方可以根据request的 id将request再归属到各自不同的服务端请求里面。
  • header压缩，如上文中所言，对前面提到过HTTP1.x的header带有大量信息，而且每次都要重复发送，HTTP2.0使用encoder来减少需要传输的header大小，通讯双方各自cache一份header fields表，既避免了重复header的传输，又减小了需要传输的大小。
  • 服务端推送（server push），同SPDY一样，HTTP2.0也具有server push功能。

多路复用和长连接复用

  • HTTP/1.* 一次请求-响应，建立一个连接，用完关闭；每一个请求都要建立一个连接；

  • HTTP/1.1 Pipeling解决方式为，若干个请求排队串行化单线程处理，后面的请求等待前面请求的返回才能获得执行机会，一旦有某请求超时等，后续请求只能被阻塞，毫无办法，也就是人们常说的线头阻塞；

  • HTTP/2多个请求可同时在一个连接上并行执行。某个请求任务耗时严重，不会影响到其它连接的正常执行；

http2.0多路复用的优点

HTTP 性能优化的关键并不在于高带宽，而是低延迟。TCP 连接会随着时间进行自我「调谐」，起初会限制连接的最大速度，如果数据成功传输，会随着时间的推移提高传输的速度。这种调谐则被称为 TCP 慢启动。由于这种原因，让原本就具有突发性和短时性的 HTTP 连接变的十分低效。
HTTP/2 通过让所有数据流共用同一个连接，可以更有效地使用 TCP 连接，让高带宽也能真正的服务于 HTTP 的性能提升。

头部压缩

假定一个页面有100个资源需要加载（这个数量对于今天的Web而言还是挺保守的）, 而每一次请求都有1kb的消息头（这同样也并不少见，因为Cookie和引用等东西的存在）, 则至少需要多消耗100kb来获取这些消息头。HTTP2.0可以维护一个字典，差量更新HTTP头部，大大降低因头部传输产生的流量。

https认证和抓包

HTTPS单向认证

先来看看单向认证的过程： 从上图可以看出，服务端拥有一对非对称密钥：B_公钥和B_私钥。详细过程如下： （1）客户端发起HTTPS请求，将SSL协议版本的信息发送给服务端。

（2）服务端去CA机构申请来一份CA证书，在前面提过，证书里面有服务端公钥和签名。将CA证书发送给客户端

（3）客户端读取CA证书的明文信息，采用相同的hash散列函数计算得到信息摘要（hash目的：验证防止内容被修改），然后用操作系统带的CA的公钥去解密签名（因为签名是用CA的私钥加密的），对比证书中的信息摘要。如果一致，则证明证书是可信的，然后取出了服务端公钥

（4）客户端生成一个随机数（密钥F），用刚才等到的服务端B_公钥去加密这个随机数形成密文，发送给服务端。

（5）服务端用自己的B_私钥去解密这个密文，得到了密钥F

（6）服务端和客户端在后续通讯过程中就使用这个密钥F进行通信了。和之前的非对称加密不同，这里开始就是一种对称加密的方式

HTTPS双向认证

双向认证和单向认证原理基本差不多，单向认证客户端需要认证服务端，而在双向认证中增加了服务端对客户端的认证 双向认证详细过程如下： （1）客户端发起HTTPS请求，将SSL协议版本的信息发送给服务端。

（2）服务端去CA机构申请来一份CA证书，在前面提过，证书里面有服务端公钥和签名。将CA证书发送给客户端

（3）客户端读取CA证书的明文信息，采用相同的hash散列函数计算得到信息摘要（hash目的：验证防止内容被修改），然后用操作系统带的CA的公钥去解密签名（因为签名是用CA的私钥加密的），对比证书中的信息摘要。如果一致，则证明证书是可信的，然后取出了服务端公钥

（4）客户端发送自己的客户端证书给服务端，证书里面有客户端的公钥：C_公钥

（5）客户端发送支持的对称加密方案给服务端，供其选择

（6）服务端选择完加密方案后，用刚才得到的C_公钥去加密选好的加密方案

（7）客户端用自己的C_私钥去解密选好的加密方案，客户端生成一个随机数（密钥F），用刚才等到的服务端B_公钥去加密这个随机数形成密文，发送给服务端。

（8）服务端和客户端在后续通讯过程中就使用这个密钥F进行通信了。和之前的非对称加密不同，这里开始就是一种对称加密的方式

ps：双向认证的客户端证书一般都可以是如openssl生成的自签名证书，包括 client.crt 和 client.key，这两部分内容可以集成在 p12 证书中, p12 证书可以设置打开密码。

HTTPS基本思路总结

HTTPS在保证数据安全传输上使用对称加密和非对称加密相结合的方式来进行的，简单来说就是通过一次非对称加密算法进行了最终通信密钥的生成、确认和交换，然后在后续的通信过程中使用最终通信密钥进行对称加密通信。之所以不是全程非对称加密，是因为非对称加密的计算量大，影响通信效率。

抓包原理

HTTPS即使安全，也是能够被抓包的，常见的抓包工具有：Charles、fildder等。

常用的HTTPS抓包方式是作为中间人，对客户端伪装成服务端，对服务端伪装成客户端。简单来说：

  • 截获客户端的HTTPS请求，伪装成中间人客户端去向服务端发送HTTPS请求
  • 接受服务端返回，用自己的证书伪装成中间人服务端向客户端发送数据内容。

反抓包策略

为了防止中间人攻击，可以使用SSL-Pinning的技术来反抓包。 可以发现中间人攻击的要点的伪造了一个假的服务端证书给了客户端，客户端误以为真。解决思路就是，客户端也检验服务端的证书，比较一下就知道真假了。

SSL-pinning有两种方式： 证书锁定（Certificate Pinning） 和公钥锁定（ Public Key Pinning）。

  • 证书锁定 需要在客户端代码内置仅接受指定域名的证书，而不接受操作系统或浏览器内置的CA根证书对应的任何证书，通过这种授权方式，保障了APP与服务端通信的唯一性和安全性，因此客户端与服务端（例如API网关）之间的通信是可以保证绝对安全。但是CA签发证书都存在有效期问题，缺点是在 证书续期后需要将证书重新内置到APP中。
  • 公钥锁定 提取证书中的公钥并内置到客户端中，通过与服务器对比公钥值来验证连接的正确性。制作证书密钥时，公钥在证书的续期前后都可以保持不变（即密钥对不变），所以可以避免证书有效期问题，一般推荐这种做法。

突破SSL-Pinning抓包

在逆向界，一山更比一山高。 思路是这样的：内置证书或者公钥的时候，常常会有对比验证的函数，直接控制这个函数的返回结果让验证通过不就好了吗。 于是就有了一个突破SLL-Pinning的经典操作：采用Xposed+justTrustme模块。 这个方案使用的是JustTrustMe这个Xposed模块，它所做的事情就是将各种已知的的HTTP请求库中用于校验证书的API都进行Hook，使无论是否是可信证书的情况，校验结果返回都为正常状态，从而实现绕过证书检查的效果。

http状态码

200：OK：表示请求成功。

204：No Content，表示服务器成功处理了请求，但没有返回任何内容。

204状态码的常见使用场景有：

DELETE请求成功删除资源后，服务器返回204表示删除成功。

PUT/POST请求被处理,但响应不包含代表资源的文档视图。

对不返回响应体的请求的成功处理。

处理OPTIONS请求后，确认请求可以处理但没有响应需要返回。

处理GET请求，但不需要返回任何状态信息。

301：Moved Permanently：永久重定向，本网页被永久性转移到另一个URL。

302：Found：临时重定向，请求的网页被转移到一个新的URL，但客户访问仍继续通过原始URL地址。

304：Not Modified：表示客户端请求的资源在服务器上没有修改，可以直接重用缓存的版本。（只返回头信息，缩短响应时间。）

304状态码的常见使用场景：

浏览器进行条件GET请求，请求头中包含”If-Modified-Since”字段，服务器判断资源未修改，返回304状态码。

文件缓存过期，浏览器发送条件请求验证文件是否修改，服务器返回304表示使用缓存文件即可。

客户端包含”If-None-Match” 字段的条件GET请求，Etag未改变，服务器返回304。

EST API条件请求资源未改变，返回304和空响应体。

400：Bad Request：客户端请求语法错误，服务器无法理解。

401：Unauthorized：请求需要用户认证。

401状态码通常在以下情况下返回：

请求需要基于用户名和密码的HTTP认证，但是请求中未提供这些信息。

提供的用户名和密码验证失败。

访问服务器的资源时使用的授权令牌无效或已过期。

资源需要客户端证书认证,但是请求中未提供证书。

用户尚未登录就访问需要登录的页面。

403：Forbidden：服务器理解用户请求，但是拒绝执行此请求。

403状态码通常在以下情况下返回：

访问服务器上的文件或目录没有足够的权限。例如访问保护目录中的文件。

没有权限访问请求的资源。例如未登录的用户请求个人信息页。

没有权限执行请求的操作。例如普通用户试图执行管理员操作。

请求的资源或操作被服务器配置为禁止访问。

服务器通过访问控制列表或类似机制，判定用户无权访问资源。

404：Not Found：服务器无法根据用户请求找到对应的资源（网页）。

404这个状态码的常见情况有：

请求的网页不存在，输入了错误的URL。

请求的文件或目录在服务器上无法找到。

请求的API接口不存在。

请求查询的对象不存在,比如不存在的 id。

访问的页面/资源已被删除。

405：Method Not Allowed：方法不允许。表示服务器已经理解请求,但是目标资源不支持该请求中所使用的方法。

常见的 405 错误产生场景：

客户端使用了服务器不支持的 HTTP 方法,比如在只支持 GET 的接口使用 POST。

服务器不允许客户端使用该方法，比如服务器禁用了 PUT 和 DELETE 方法。

临时禁用了某个方法的使用，比如网站维护时禁用POST提交。

注：可以检查Allow头并使用允许的方法重试。

408：Request Timeout：请求超时。当客户端发起一个请求，但在服务器预期的时间内没有收到请求的完整信息。

常见的 408 产生原因有：

客户端请求阻塞,请求体发送速度过慢。

客户端请求体过大,在 server 定义的超时时间内无法完成。

网络连接速度不稳定,请求中断。

客户端断开连接,服务器等待请求完成超时。

500：Internal Server Error：服务器内部错误，服务器端代码错误。

一些常见触发500错误的情况：

服务器未捕获的代码异常。

服务器数据库查询错误。

服务器无法处理合法的请求。

服务器资源不足错误。

服务器配置错误导致无法完成请求。

代理服务器接收到无效响应。

服务器维护升级时发生内部错误。

502：Bad Gateway：作为网关或代理工作的服务器尝试执行请求时，从上游服务器接收到了一个无效的响应。

一些常见的502错误情况：

网关服务器尝试执行请求时，上游服务器返回了无效的响应。

代理服务器试图连接上游服务器，但收到了非预期的响应。

网关服务器未收到上游服务器的及时响应。

代理服务器有路由问题，不能选择正确的上游服务器。

网关服务器因上游问题无法访问到正确的响应。

代理软件配置错误,无法连接上游服务器。

503：Service Unavailable：服务器当前无法处理请求,这通常是一个临时的状态。（通常会在HTTP响应头中包含一个”Retry-After”字段，指明多久后服务可能恢复。）

返回503状态码的常见情况：

服务器过载或维护，暂时的无法处理请求。

服务器依赖的其他服务暂时不可用。

服务器正在进行停机维护。

服务器由于达到资源使用上限而拒绝服务。

服务器启用安全或防火墙规则而拒绝访问。

服务器上的程序崩溃了，需要重启。

504：Gateway Timeout：网关超时。网关服务器(代理服务器)作为客户端向后端服务器发出请求，后端服务器长时间未响应。

504的常见情况：

后端服务器处理缓慢,超过网关的超时时间。

后端服务器发生故障,完全无响应。

网关服务器和后端服务器之间的网络出现问题,请求无法到达后端服务器。

后端服务器依赖的外部服务(比如数据库)响应缓慢或失效,导致整体响应超时。

网关服务器的超时时间设置过短,不能容忍后端的正常波动。

后端服务器负载过高,单个请求处理时间变长,超过了网关的超时配置。

后端服务器或者网关服务器的Bug,导致某些请求处理卡死。

OPTIONS请求在实际中的作用

  • 查看Web服务器的性能，可以测试不同的选项来判断服务器的性能。
  • 查看服务器支持的HTTP方法，在实际发送请求前，利用OPTIONS请求来检查服务器支持的方法。
  • CORS预检请求，先发送OPTIONS请求来判断跨域请求是否安全，服务器需要返回正确的CORS头 Signal preflight request.
  • 灰度测试，可以先用OPTIONS请求检测新功能是否可用,再确定是否调用。

OPTIONS请求主要的作用是让客户端安全地发现服务器的能力，进行功能和兼容性测试。它比直接发送请求更安全可靠。

跨域资源共享CORS

CORS（Cross-Origin Resource Sharing）是HTTP协议对浏览器中不同网站间AJAX请求的规范和限制。

AJAX即“Asynchronous Javascript And Xml”(异步JavaScript和XML)，是指一种创建交互式网页应用的网页开发技术。AJAX是一种浏览器通过js异步发起请求，局部更新页面的技术。AJAX请求的局部更新，浏览器地址栏不会发生变化，局部更新不会舍弃原来页面的内容。

通过$http_origin获取HTTP请求头中的Origin字段的值，验证请求来源是否在允许的域列表中。

CORS原理：

网站是部署在服务器某个端口上的应用进程，通过监听端口来接收外界的访问 ，外界通过域名（IP地址）来找到对应服务器，通过协议/端口来找到对应应用进程。当某个网站需要访问其他网站的资源时，就会发生跨域请求。而浏览器会根据CORS协议来核实请求是否被目标网站所允许。

第一步，浏览器预处理，包含请求头信息：Host（目标网站域名）和Origin（源网站域名）。

第二步，判断是简单请求还是复杂请求。如果是简单请求，走第三步。如果是复杂请求，走第四步。

第三步，向目标网站发送资源请求。走第七步。

第四步，向目标网站发送预检请求（OPTIONS）。

第五步，服务器收到预检请求后，判断Origin是否在白名单中，如果是，则包含响应头Access-Control-Allow-Methods、Access-Control-Allow-Headers等信息，不带任何响应内容，状态码是204。

第六步，浏览器验证服务器响应头，复杂请求是否允许，如果允许，走第三步，否则，打印错误信息。

第七步，服务器收到资源请求后，判断Origin是否在白名单中，如果是，则包含响应头Access-Control-Allow-Methods、Access-Control-Allow-Headers等信息，并返回对应资源内容。（如果服务器为反向代理服务器，会先转发请求到后端服务器，之后再在后端服务的响应中添加响应头）

第八步，浏览器收到目标网站的响应，验证响应头，如果是正确响应，则取出数据，否则，打印错误信息。
